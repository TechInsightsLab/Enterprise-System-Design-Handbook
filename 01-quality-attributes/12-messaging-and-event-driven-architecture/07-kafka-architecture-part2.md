
# 51. Kafka Log Segments

> **A Kafka partition is not stored as one giant file. Instead, it is divided into multiple smaller log segment files.**

This is one of the reasons Kafka can efficiently store:

- Billions of messages
- Terabytes of data
- Long retention periods

---

# Why Log Segments?

Imagine a topic receives:

```
500 Million Messages
```

Storing everything in:

```
One Huge File
```

creates problems.

- Slow recovery
- Slow deletion
- Difficult maintenance

Instead,

Kafka splits logs.

---

# Architecture

```text
Partition

↓

Segment 1

↓

Segment 2

↓

Segment 3

↓

Segment 4
```

Each segment is an independent file.

---

# Example

```text
Partition 0

├── 000000000000.log

├── 000000100000.log

├── 000000200000.log

└── 000000300000.log
```

Each file stores a range of offsets.

---

# Segment Rotation

Kafka creates a new segment when:

- File size exceeds limit
- Time threshold reached

Example.

```
1 GB

↓

New Segment
```

or

```
24 Hours

↓

New Segment
```

---

# Benefits

- Faster cleanup
- Efficient retention
- Easier replication
- Better recovery
- Reduced maintenance cost

---

# 52. Kafka Retention Policy

> **Retention determines how long Kafka keeps messages after they are written.**

Unlike traditional queues,

Kafka does **not** immediately delete messages after consumption.

Messages remain until retention rules remove them.

---

# Why?

Consumers may need to:

- Replay events
- Recover failures
- Build new systems
- Reprocess history

---

# Example

Order Event.

```
OrderPlaced
```

Consumer processes.

Message remains.

```
7 Days
```

New consumer joins.

Reads old events.

Possible.

---

# Time-Based Retention

Most common.

Example.

```
7 Days
```

After.

```
7 Days
```

Segment removed.

---

# Size-Based Retention

Example.

```
500 GB
```

Oldest segments removed.

When storage exceeds:

```
500 GB
```

---

# Combination

Kafka may use both:

```text
Time

AND

Size
```

Whichever limit is reached first.

---

# Retention Examples

| Workload | Typical Retention |
|-----------|-------------------|
| Audit Events | Months to Years |
| User Activity | Days to Weeks |
| Metrics | Hours to Days |
| CDC | Days |
| Financial Events | Depends on regulatory requirements |

Retention is a business and compliance decision.

---

# Benefits

- Replay capability
- Recovery
- Analytics
- New consumer onboarding

---

# Trade-offs

Longer retention.

Requires.

More storage.

---

# 53. Log Compaction

> **Log Compaction retains only the latest value for each key while preserving the complete history for keys that have not been updated.**

Unlike retention,

compaction is:

Key-based.

---

# Example

Customer.

```
Customer101

↓

Version1

↓

Version2

↓

Version3
```

Compaction keeps.

```
Version3
```

Older versions.

Eventually removed.

---

# Why?

Configuration.

Latest value.

Matters.

Not every historical update.

---

# Example

Feature Flag.

```
FeatureX

↓

Enabled

↓

Disabled

↓

Enabled
```

Only latest state.

Needed.

---

# Retention vs Compaction

| Retention | Compaction |
|------------|------------|
| Removes old messages by age/size | Removes obsolete versions by key |
| Time-based | Key-based |
| Deletes complete segments | Keeps latest record per key |

---

# Enterprise Uses

- Customer Profiles
- Product Catalog
- Configuration
- CDC
- Reference Data

---

# 54. Producer Idempotence

> **Producer idempotence ensures that retries do not create duplicate records in Kafka partitions.**

Imagine.

Producer sends.

```
Order100
```

Network timeout.

Producer retries.

Without protection.

Two identical messages.

---

# Workflow

Without Idempotence.

```text
Producer

↓

Broker

↓

Timeout

↓

Retry

↓

Duplicate Message
```

---

With Idempotence.

```text
Producer

↓

Broker

↓

Retry

↓

Duplicate Ignored
```

---

# Benefits

- No duplicate writes caused by producer retries
- Better reliability
- Safer retries

---

# Important Note

Producer idempotence prevents duplicates **generated by producer retries**.

Consumers should still be designed to be idempotent because duplicates can arise from other failure scenarios.

---

# 55. Kafka Transactions

> **Kafka transactions allow multiple writes to be committed atomically across partitions and topics.**

Without transactions.

Imagine.

Write.

```
Order Event

Success
```

Write.

```
Inventory Event

Failed
```

System inconsistent.

---

# Transaction Workflow

```text
Begin Transaction

↓

Write Event A

↓

Write Event B

↓

Commit
```

Either.

Everything succeeds.

Or.

Nothing visible.

---

# Enterprise Uses

- Financial Events
- Event Publishing
- Outbox Integration
- Stream Processing

---

# Trade-off

Transactions improve consistency.

But increase:

- Latency
- Coordination
- Operational complexity

# 56. Consumer Rebalancing

> **Rebalancing redistributes partitions among consumers when group membership changes.**

---

# Example

Initial.

```text
Consumer1

↓

P0

Consumer2

↓

P1
```

New consumer joins.

---

After rebalance.

```text
Consumer1

↓

P0

Consumer2

↓

P1

Consumer3

↓

P2
```

Workload redistributed.

---

# When Rebalancing Happens

- New consumer joins
- Consumer leaves
- Consumer crashes
- Partition count changes

---

# Why?

Maintain.

Balanced workload.

---

# Drawback

During rebalance.

Consumers briefly pause processing.

Frequent rebalances can impact throughput.

---

# Minimizing Rebalances

- Stable consumer groups
- Static membership (where supported)
- Cooperative rebalancing
- Appropriate session timeout settings

---

# 57. Consumer Lag

> **Consumer Lag is the difference between the latest produced offset and the last processed offset.**

It measures:

How far consumers are behind.

---

# Example

Latest Offset.

```
1000
```

Consumer Offset.

```
900
```

Lag.

```
100 Messages
```

---

# Why Lag Matters

Growing lag means.

Consumers cannot keep up.

Possible causes.

- Slow processing
- Traffic spike
- Consumer failures
- Insufficient consumers

---

# Monitoring

Important metrics.

- Current Lag
- Maximum Lag
- Lag Growth Rate

---

# Reducing Lag

- Add consumers
- Optimize processing
- Increase partitions
- Improve downstream systems

---

# Example

Traffic.

```
1 Million Events

Per Minute
```

Consumers.

```
500,000

Per Minute
```

Lag.

Continues growing.

# 58. Kafka Performance Optimization

Architects optimize Kafka by tuning.

- Partitions
- Batch size
- Compression
- Replication
- Producer acknowledgements
- Consumer concurrency

---

# Producer Batching

Instead of sending.

One message.

At a time.

Batch.

```
100 Messages
```

Together.

---

# Benefits

- Fewer network calls
- Higher throughput
- Lower CPU usage

---

# Trade-off

Larger batches.

Slightly higher latency.

---

# Compression

Kafka supports compression.

Examples.

- GZIP
- Snappy
- LZ4
- Zstandard (Zstd)

---

Benefits.

- Lower bandwidth
- Lower storage

Trade-off.

Higher CPU usage.

---

# Choosing Compression

| Algorithm | Characteristics |
|------------|----------------|
| GZIP | High compression, higher CPU |
| Snappy | Fast compression/decompression |
| LZ4 | Very fast, balanced |
| Zstd | Excellent compression with good performance |

Choice depends on workload.

---

- Operational overhead

# Broker Scaling

Need more throughput?

Options.

- Add brokers
- Increase partitions
- Optimize producers
- Optimize consumers

---

# Throughput vs Latency

Higher throughput often comes from:

- Larger batches
- More partitions
- Compression

Lower latency often requires:

- Smaller batches
- Faster acknowledgements
- Lower queue depth

Architects balance both.

---

# Common Mistakes

### One Partition for High Traffic

Creates a bottleneck.

Parallelism impossible.

---

### Thousands of Tiny Partitions

Excessive metadata and longer rebalances.

Partition count should reflect realistic throughput needs.

---

### Unlimited Retention

Storage grows indefinitely.

Define retention based on business and compliance requirements.

---

### Ignoring Consumer Lag

Lag is one of the earliest indicators of processing problems.

Alert on sustained lag growth.

---

### Enabling Transactions Everywhere

Transactions are powerful.

But unnecessary for many event streams.

Use only where business consistency requires them.

---

# Architect's Perspective

Kafka is fundamentally an **append-only distributed log**.

Everything else—

- Replay
- Consumer groups
- Event streaming
- Scalability
- Fault tolerance

is built on that simple concept.

Successful Kafka architectures are designed around four key decisions:

1. **Partition strategy** — How is work distributed?
2. **Retention strategy** — How long should events be kept?
3. **Replication strategy** — How much durability is required?
4. **Consumer strategy** — How will events be processed and scaled?

When these decisions align with business requirements, Kafka becomes one of the most scalable and resilient event streaming platforms available.
