
# 5. Metrics Deep Dive

> **Metrics are numerical measurements collected over time that describe the health, performance, reliability, and business behavior of a system.**

Metrics are the most commonly used observability signal because they are:

- Lightweight
- Efficient
- Easy to aggregate
- Excellent for dashboards
- Excellent for alerting

Nearly every production dashboard is primarily built using metrics.

---

# Why Metrics Matter

Imagine an online shopping platform.

Without metrics.

You cannot answer questions like:

- How many users are active?
- Is checkout getting slower?
- Are payment failures increasing?
- Is CPU utilization rising?
- Is cache performance degrading?

Metrics answer these questions continuously.

---

# Metrics Lifecycle

```text
Application

↓

Instrumentation

↓

Metric Generated

↓

Collector

↓

Time-Series Database

↓

Dashboard

↓

↓

```

Metrics flow continuously through the observability pipeline.

---

# Characteristics of Good Metrics

Metrics should be:

- Numeric
- Timestamped
- Aggregatable
- Low Cardinality
- Meaningful
- Consistently Named

Poor metric design leads to higher storage costs and slower queries.

---

# Metric Components

Every metric consists of:

```text
Metric Name

↓

Value

↓

Timestamp

↓

Labels (Dimensions)
```

---

# Example

```text
http_requests_total

Value

2500

Timestamp

10:15

Labels

service=order

method=POST

status=200
```

Labels provide context.

---

# Labels (Dimensions)

Labels enrich metrics.

Example.

```text
http_requests_total

service=payment

region=india

status=200

method=POST
```

Instead of creating multiple metrics,

one metric can answer many questions.

---

# Why Labels Matter

Without labels.

Need separate metrics.

```text
payment_requests

inventory_requests

order_requests
```

With labels.

One metric.

```text
http_requests_total

service=payment

service=inventory

service=order
```

Much simpler.

---

# Cardinality

> **Cardinality refers to the number of unique label combinations generated by a metric.**

---

# Low Cardinality

Good.

```text
status

200

400

500
```

Only three values.

---

# High Cardinality

Bad.

```text
customerId

1

2

3

...

10 Million
```

Every customer.

Creates new time series.

Storage explodes.

---

# Avoid High Cardinality Labels

Avoid using:

- User ID
- Email
- Session ID
- Order ID
- Transaction ID

Prefer.

- Region
- Status
- Service
- Environment
- API Name

---

# Metric Naming

Good names.

```text
http_requests_total

payment_latency_seconds

cache_hit_ratio

database_connections
```

Bad names.

```text
metric1

abc

counter_new
```

Naming consistency is critical for enterprise observability.

---

# Standard Metric Units

Use standard units.

Examples.

```text
Milliseconds

Seconds

Bytes

Percentage

Requests

Errors
```

Avoid inventing custom units.

---

# Types of Metrics

Modern observability platforms commonly use four primary metric types.

```text
Metrics

├── Counter
├── Gauge
├── Histogram
└── Summary
```

Each serves a different purpose.

---

# 6. Counter

> **A Counter is a cumulative metric that only increases over time (except when reset due to process restart).**

Counters represent.

Events.

That happen.

Examples.

- Requests
- Orders
- Errors
- Logins
- Payments

---

# Example

```text
Requests

↓

100

↓

101

↓

102

↓

103
```

Never decreases.

During normal execution.

---

# Good Use Cases

- Total Requests
- Total Errors
- Orders Created
- Kafka Messages
- Login Attempts

---

# Example Metrics

```text
http_requests_total

orders_created_total

payments_processed_total

login_failures_total
```

---

# What Not to Use Counter For

Current CPU.

Wrong.

Current Memory.

Wrong.

Current Queue Size.

Wrong.

These values go up.

And down.

Use Gauge.

---

# 7. Gauge

> **A Gauge represents a value that can increase or decrease over time.**

Unlike counters.

Gauges.

Move freely.

---

# Example

CPU Usage.

```text
65%

↓

72%

↓

45%

↓

88%
```

---

# Good Use Cases

- CPU Usage
- Memory Usage
- Queue Size
- Active Sessions
- Database Connections
- JVM Heap

---

# Example Metrics

```text
cpu_usage

memory_usage

active_users

db_connections
```

---

# Gauge Visualization

```text
CPU

70%

↓

45%

↓

82%

↓

61%
```

Continuously changing.

---

# Counter vs Gauge

| Counter | Gauge |
|----------|--------|
| Only Increases | Increases & Decreases |
| Total Requests | CPU Usage |
| Total Orders | Memory Usage |
| Total Errors | Queue Length |

---

# 8. Histogram

> **A Histogram measures the distribution of values by grouping observations into configurable buckets.**

Histograms answer.

Questions like.

```
How many requests completed within:

10 ms?

100 ms?

500 ms?

1 second?
```

---

# Example

Request Latency.

```text
0-100 ms

450 Requests

------------------

100-500 ms

300 Requests

------------------

500-1000 ms

25 Requests
```

---

# Buckets

Buckets divide.

Measurements.

Into ranges.

Example.

```text
10 ms

50 ms

100 ms

250 ms

500 ms

1 sec

2 sec
```

---

# Why Histograms?

Average latency.

Can hide.

Problems.

Example.

```text
Average

100 ms
```

Looks good.

But.

```text
95%

↓

20 ms

5%

↓

3 Seconds
```

Histogram reveals.

Outliers.

---

# Enterprise Uses

Histograms commonly measure:

- API Latency
- Database Query Time
- File Upload Duration
- Kafka Publish Time
- Cache Lookup Time

---

# Percentiles

Histograms enable percentile calculations.

Examples.

- P50
- P90
- P95
- P99

Widely used.

For SLOs.

---

# 9. Summary

> **A Summary records observations and calculates quantiles (such as P95 or P99) on the client side before exporting telemetry.**

Summaries appear similar.

To histograms.

But differ.

Internally.

---

# Example

Summary.

Exports.

```text
Average

P95

P99

Count

Sum
```

---

# Histogram vs Summary

| Histogram | Summary |
|------------|----------|
| Buckets | Quantiles |
| Aggregatable Across Instances | Not Easily Aggregated |
| Preferred for Large Distributed Systems | Good for Single Instance Analysis |

---

# Enterprise Recommendation

In distributed systems,

histograms are generally preferred because they can be aggregated across multiple instances.

# Common Mistakes

### Using Gauge Instead of Counter

Total requests.

Should never decrease.

Use counters.

---

### High Cardinality Labels

Avoid labels such as:

- User ID
- Email
- Session ID

They dramatically increase storage and query costs.

---

### Measuring Everything

Thousands of unnecessary metrics.

Increase costs.

Without adding value.

Measure what matters.

---

### Inconsistent Naming

Enterprise teams should establish naming conventions.

Before production deployment.

---

### Ignoring Business Metrics

Technical metrics explain.

System health.

Business metrics explain.

Business health.

Architects need both.

---

# Architect's Perspective

Metrics are the foundation of operational visibility.

A mature observability strategy includes metrics at three levels:

1. **Infrastructure Metrics** — Is the platform healthy?
2. **Application Metrics** — Is the software healthy?
3. **Business Metrics** — Is the business healthy?

Architects should design metrics that answer business questions, not merely technical ones.

A healthy server is meaningless if:

- Orders cannot be placed.
- Payments are failing.
- Customers are abandoning checkout.

Ultimately, the most valuable metrics are those that connect **technical performance** with **business outcomes**.
